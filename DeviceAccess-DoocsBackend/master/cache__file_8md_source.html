1 CacheFile Mechanism {#cacheFile}
2 ===================
3 
4 This feature is a workaroud to deal with long RegisterCatalogue fetch times for Doocs locations with huge property
5 lists. It requires a description file to be provided to the backend.
6 
7 To enable this feature, specify the cacheFile parameter in the applications %ChimeraTK Device Descriptor (**cdd**):
8 
9 \verbatim
10 (doocs:XFEL.RF/TIMER/LLA6M?cacheFile=Filename.xml)
11 \endverbatim
12 
13 Filename.xml describes the Doocs locations property list. An example file may look like this:
14 
15 \verbatim
16  <?xml version=\"1.0\" encoding=\"UTF-8\"?>
17  <catalogue version=\"1.0\">
18  <register>
19  <name>/DUMMY</name>
20  <length>1</length>
21  <access_mode></access_mode>
22  <doocs_type_id>1</doocs_type_id>
23  <!--doocs id: INT-->"
24  </register>"
25  </catalogue>";
26 \endverbatim
27

29 ## Using DoocsBackend to Generate Xml Descriptor File
30 The descriptor xml file may be user generated. It is also possible to make the DoocsBackend generate one. The steps below detail
31 the process:
32 - Set cacheFile key in the cdd to the desired xml filename (eg. Filename.xml).
33 - Start the server application and run it till Filename.xml is generated in the working
34  directory.
35 - Exit server application.
36 
37 ## Behaviour Specifics:
38 1. The DoocsBackend always starts a background task for querying the Doocs location and creating a RegisterCatalogue
39  from this information.
40 2. When a cacheFile xml is provided in the cdd, it is parsed to generate the RegisterCatalogue. The RegisterCatalogue
41  fetched by background task in step.1 remains unused (except to replace the xml file on disk - see step.3).
42 3. In cases where cacheFile parameter is specified in cdd:
43  - cacheFile xml on disk is replaced with the RegisterCatalogue info from step.1, once this background task
44  completes.
45  - cacheFile xml is not modified if server application exits before task in step.1 is complete.
46 4. If no cacheFile parameter is provided in the cdd, backend waits for the task in step.1 to complete. The generated
47  RegisterCatalogue from this task is then used by the application, but will not be cached to disk as an xml file.

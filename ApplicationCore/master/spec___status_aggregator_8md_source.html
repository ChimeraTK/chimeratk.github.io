10 ## Introduction ##
11 
12 The `ChimeraTK::StatusAggregator` provides a facitlity to collect multiple
13 statuses and merge them into a single output status. The sources of input statuses can be:
14 
15 * Instances of `ChimeraTK::StatusMonitor`
16 * Other instances of the StatusAggregator
17 
18 ## High-level requirements ##
19 
20 ### R1.1 Automatic detection of instances ###
21 
22 The StatusMonitor and StatusAggregator instances to be connected into the
23 StatusAggregator should be found automatically as follows:
24 
25 * **R1.1.1**: A StatusAggregator aggregates all instances in the hierarchy of
26  ApplicationCore Modules which are at the same hierarchy level or below
27 * **R1.1.2**: Two StatusAggregator instances cannot be on the same hierarchy
28  level - this should throw a `ChimeraTK::logic_error`
29 * **R1.1.3**: If a StatusAggregator is found at some hierarchy level below, the
30  search is stopped for that branch of the hierarchy tree, i.e. all variables
31  which are already aggregated by that aggregator are not aggregated again
32 * **R1.1.4**: The StatusAggregator provides a `disable` input, which
33  deactivates the aggregator and sets the output status to OFF.
34 
35 ### R1.2 Priorization of status values ###
36 
37 The StatusAggregator is configurable (by a constructor parameter, not a PV) in
38 the way how the statuses are prioritised. The status with the highest priority
39 within the aggregated instances defines the aggregated status. The following
40 options exist (highest priority first):
41 
42 1. error - warning - off - ok
43 2. error - warning - ok - off
44 3. error - warning - ok or off - mixed state of ok/off results in a warning
45 4. off - error - warning - ok
46 
47 ### Constraints and issues ###
48 
49 * As detection of underlying instances can only be performed in the
50  constructor, the StatusAggregator has to be declared in user code after all
51  instances of `ChimeraTK::StatusMonitor` in a Module.
52 * Usage of `HierarchyModifier::moveToRoot` on a StatusAggregator is
53  controversial (currently not allowed, see R2.2):
54  * Either: Detection of aggregated instances may take place from the virtual
55  hierarchy level of the original (C++) parent module downwards. The
56  Aggregator and its status output would then appear on the root level.
57  * Or: `HierarchyModifier::moveToRoot` is not allowed on StatusAggregators
58 
59 ## Implementation ##
60 
61 ### Requirements ###
62 
63 * **R2.1**: The detection of instances needs to be performed recursively from the StatusAggregator's virtual parent module on
64 * **R2.2**: The detection has to work on instances the are modified by `ChimeraTK::HierarchyModifier`s `none` and `hideThis`. Other modifiers should give a `ChimeraTK::logic_error`
65 
66 ### Constraints and issues ###
67 
68 * The detection and connection of the instances to be aggregated has to be
69  performed in the constructor, later the variable household is fixed and the
70  status inputs can not be added anymore.
71 
72 * As a consequence of the implementation requirements, in order to get the
73  virtual parent module, the entire Application needs to be virtualised and the
74  search algorithm must then navigate to the parent module.
75 
76 * Related to R1.1.3: If a StatusAggregator is found on a level below, the
77  instances of `ChimeraTK::StatusMonitor`s on that level must be ignored.
